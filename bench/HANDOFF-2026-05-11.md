## The three failure modes you'll see (and what they mean)

### Class A: vision-model hallucination (Run 1)

Brain describes a UI element that isn't actually on screen. Grounder
fabricates click coords. Coords scatter across 250px+ between
attempts because there's no real target. **Anti-loop guard
`coord-scatter`** (commit `4956638`) catches this — bails after 3
scattered grounds on the same action text.

Mitigation: the TASK PRIORITY preamble explicitly tells the brain
"if you described it and the screen didn't change, the element
isn't there — don't emit the description a third time".

### Class B: task-ordering failure (Run 2)

Brain looks at the screen, ignores the prompt's first instruction,
and acts on what's visible instead. Run 2: brain tried to close
Chrome tabs instead of opening Excel (Excel was the prompt's first
instruction). Same coord 3× with no screen change. **Anti-loop guard
`no-op-spam`/`same-action`** catches this.

Mitigation: TASK PRIORITY preamble rule #1 — "follow task order;
if the task says 'first open X' and X isn't visible, your first
action is to open it; don't tidy up unrelated visible state".

### Class C: false-positive DONE (Run 3, Ponder UI path)

Brain emits a confident success message for work it never did.
Verifier (or planner+router stack on UI path) approves it. **This
is the OSWorld "lying about completion" failure class.** The
strict verifier (`1957823`) was supposed to eliminate this for the
bench-harness path but doesn't catch it on the Ponder UI path.

Mitigation: **not yet shipped**. Candidates:
- Tighten the local router (`src/agent/router.ts`) to NOT match
  unrelated patterns — it pattern-matched "Honda CR-V research"
  → mathway.com (a calculator URL from earlier sessions in the
  same Ponder Modal session).
- Force strict-verifier on the Ponder UI path too.
- Make the bench harness the ONLY supported entrypoint for benchmarks
  (the UI path can stay for interactive use).

## The harness contract (read this before adding cases)

Every case `.md` has:

```markdown
---
task_id: <slug-matching-filename>
tier: T1 | T2 | T3 | T4 | T5
axes:
  - <one of the 10 difficulty axes from SUITE.md>
expected_actions: <int>
max_steps: <int, optional — overrides MAX_STEPS=50 default>
infeasible: false | true
---

# <Human-readable title>

## Task (verbatim prompt)

> The exact natural-language prompt sent to agent_do. Use blockquote
> lines; harness joins them with single spaces.

## Setup

```sh
# Bash script. Env vars available: RUN_START_ISO.
# Non-zero exit doesn't abort (intentional — some setups have probes).
```

## Scoring

```sh
# Bash script. Env vars available: OUTCOME, RUN_START_ISO, RUN_END_ISO.
# Must emit a final line starting with "PASS" or "FAIL". The harness
# parses the LAST such line as the verdict.
# Example outputs:
#   PASS: 3 valid listing rows
#   FAIL: no spreadsheet at /Users/.../...
#   FAIL: CATASTROPHIC — fixture file missing
```
```

The harness writes `bench/results/<case-id>-<iso>.json` with BOTH
`deterministic_score` AND `agent_self_reported_outcome` AND a
`disagreement: bool` flag when they differ. Authority order:
deterministic > verifier > self-report.

Large screenshots are auto-stripped to sidecar `*-final.png` files —
result JSONs stay under 10 KB.

## What's pending (in priority order for next session)

### 1. Post-Spotlight screenshot timing race (TOP PRIORITY)

Run 4 (bench Run 3 with TASK PRIORITY preamble) revealed the next
failure boundary:

```
[t=8.2s]  cmd+space           ← correct
[t=15.6s] type "Microsoft Excel" + enter   ← correct
[t=25.3s] click Google Chrome to foreground   ← WRONG — Excel was still launching
[t=30.4s] click {207, 151}
... etc, brain regressed to fighting Chrome
```

The screenshot taken between step 2 and step 3 still showed Chrome
because Excel hadn't appeared in the frame yet (either still loading,
or window not yet receiving focus). The brain reacted to Chrome as if
the Spotlight launch failed and went back to fighting tabs.

Three fix candidates (pick one):

a) **Post-action settle delay** for "launch app" actions. When the
   executor sees `cmd+space` followed by `type X + enter`, treat that
   as a "launch sequence" and inject a 1500-3000ms sleep before the
   next screenshot capture. Implementation: ~15 LOC in
   `src/agent/loop.ts` around the screenshot-after-action capture
   site. Risk: heuristic — might over-wait when Spotlight is used for
   non-app queries.

b) **A `wait_for_app <name>` action verb.** Brain emits e.g.
   `wait_for_app Microsoft Excel`; executor polls `osascript -e 'tell
   application "System Events" to get name of first application process
   whose frontmost is true'` for up to 3s. Cleanest but needs:
   - Add `wait_for_app` to the action verb allow-list in `brain.ts`
   - Executor branch in `loop.ts` to handle it
   - Brain prompt note: "after launching an app via Spotlight, your
     next action should be `wait_for_app <name>`"
   ~30 LOC total.

c) **Brain prompt rule**: add to the TASK PRIORITY preamble or as a
   separate block: "After you launch an app via Spotlight (cmd+space
   + type name + enter), the next screenshot may briefly still show
   the previous app — the launched app needs ~1-2 seconds to appear.
   Don't react to the previous app being visible; emit `wait 2s` for
   one step, then re-observe." Cheapest fix (~5 LOC) but relies on
   the small model to follow a conditional rule. Bench-evidence
   suggests it'll work given the preamble's success on the launch
   sequence.

I'd start with (c) — cheapest, fastest to validate, falls back to
(a) or (b) if it doesn't move the needle.
