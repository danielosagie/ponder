# v1.2 Plan: hosted Ponder Cloud + multi-tenancy + monetization

This document is a single-source design for everything between the current shipped state (v1.1, items 1+5+6+7) and a charging, multi-tenant, optionally-hosted SaaS (v1.2). It folds in eight items previously marked out-of-scope plus the original item 8 ("Hosted Ponder Cloud").

The plan is opinionated. It picks a sequence, calls out the architectural decisions that lock in early, and flags the open questions that need a real answer before code lands.

---

## 1. Goals (what "v1.2 done" means)

A customer can:
1. Sign up at `cloud.ponder.dev` with an email
2. Install Ponder.app on their Mac (BYO worker — the default and free tier) **or** spin up a hosted Linux worker from the dashboard (paid tier)
3. Get a `convex` URL + API key wired automatically to their workspace
4. Dispatch tasks via `PonderClient`; the right runtime claims them
5. See live + historical session steps in the dashboard
6. Get billed monthly via Stripe metered pricing on `usageRollups`

Out of scope for **v1.2** (deferred to v2 or beyond):
- Plugin / extension system for custom action verbs (item 8 in the original out-of-scope list)
- Sub-second multi-worker cancel propagation (we'll ship "next-await-boundary" cancel)
- Multi-region Modal deployments

These three are explicitly punted because each adds weeks of work and none unblocks the "charge customers money" path.

## 2. Architecture: the hybrid worker model

You picked "BYO default, hosted as add-on." The architecture has to make hosted workers a clean drop-in, not a retrofit.

### 2.1 Worker abstraction

A worker — BYO or hosted — is whatever speaks the existing fleet protocol from item 7:
- `workers.register({ workerId, hostname, platform, capabilities })`
- `workers.heartbeat({ workerId, status })` every 5s
- `workers.claimNext({ workerId, runtime })` every 5s, transitions a session pending→running
- `workers.releaseSession({ workerId, sessionId, status })`
- `workers.goOffline({ workerId })` on shutdown

The desktop already implements this. A hosted worker is a Linux container running the **same Electron app** plus a display-streaming sidecar. From Convex's perspective it's just another row in `workers`.

### 2.2 New capability: `hosted-linux`

Extend `workers.capabilities` to a wider union:

```ts
capabilities: v.array(
  v.union(
    v.literal("desktop"),         // BYO Mac — claims runtime "desktop" sessions
    v.literal("headless"),        // serveHeadless() consumer — already in v1.1
    v.literal("hosted-linux"),    // v1.2 — Ponder-managed Linux container
  ),
),
```

And extend `sessions.runtime` symmetrically:

```ts
runtime: v.optional(
  v.union(
    v.literal("desktop"),
    v.literal("headless"),
    v.literal("hosted"),          // v1.2 — wants a Ponder-hosted worker
  ),
),
```

The `claimNext` filter from `convex/workers.ts` already does `runtime === args.runtime`. No code change needed there — just the value enum widens.

### 2.3 Display streaming for hosted workers

**Decision: Selkies-gstreamer (WebRTC) with a TURN-server fallback.** WebRTC latency matters because the customer is *watching* the agent click — laggy video is the difference between "magical" and "cursed." Selkies ships ready-made Docker images, integrates with Pion for the signaling, and gives ~50-100ms RTT on US-East. We run a single coturn instance for NAT-restrictive networks (cheap; ~$15/mo on a t4g.small).

The customer never SSHs in. They click "Watch live" in the dashboard, which opens a session-scoped WebRTC stream gated by a short-lived JWT. The stream is 1:1 with `sessions._id`, terminated when the session moves to `done | error | cancelled`.

Selkies wants GPU hardware encoding for best quality. AWS Fargate doesn't expose GPUs, so we use **x264 software encoding at 720p / 30fps** — the agent's UI is mostly text + simple animations, so motion compression is easy. CPU overhead is ~30% of one vCPU. Sufficient quality test: text in Notes.app is sharp; YouTube videos in the agent's browser are intentionally not the primary use case.

### 2.4 Hosted-worker lifecycle

Per workspace, we run a small pool of always-warm Linux containers (default 1, paid tiers can scale to N). Each container:

- Has `WORKSPACE_ID` baked into its env
- On boot: registers as a worker with `capabilities: ["hosted-linux"]`, `workspaceId` set
- Runs the same `electron/main.ts` we already ship
- Exposes a Selkies endpoint at `/stream/:sessionId` gated by a session-scoped JWT

When a customer hits "Spin up hosted worker" in the dashboard, we call `fly.io machines run` (or AWS Fargate, or Modal — see §6) with the workspace's image. The container registers itself. From the SDK's side, dispatching with `runtime: "hosted"` just works.

**Cold-start tradeoff:** zero-warm pool = $0 idle but 30-60s startup; one-warm = ~$8/mo per workspace. Default to zero-warm on the free tier (BYO only — no hosted at all), one-warm on paid tiers.

## 3. Per-customer auth (out-of-scope #1 → in scope)

This is the foundation. Without it, items 8 + #3 (Stripe) can't safely multi-tenant.

### 3.1 Identity provider

**Decision: Clerk.** Drop-in B2B SaaS auth; built-in orgs/workspaces; Convex integration via JWT template; pre-built `<UserButton/>` and `<OrganizationSwitcher/>`. Cost is ~$25/mo + per-MAU on top; we eat that as a customer-acquisition expense in exchange for shaving ~2 weeks off the auth build (we'd otherwise need to model orgs ourselves on top of Convex Auth).

Migration path off Clerk later (Convex Auth, WorkOS, whatever) is mechanical — the JWT verifier shape is the same.

### 3.2 Schema additions

```ts
// New table — one row per Clerk org. We don't duplicate user identity; we
// just key off Clerk's org ID.
customers: defineTable({
  clerkOrgId: v.string(),                  // "org_2ab..."
  email: v.string(),                       // billing contact
  planTier: v.union(
    v.literal("free"),
    v.literal("starter"),
    v.literal("growth"),
  ),
  stripeCustomerId: v.optional(v.string()),
  stripeMeterId: v.optional(v.string()),   // for metered billing (§5)
  hostedQuota: v.number(),                 // max concurrent hosted workers
  createdAt: v.number(),
}).index("by_clerkOrg", ["clerkOrgId"]),

// Add customerId to existing tables. We keep workspaceId as-is so deployments
// without Clerk (self-host) still work; for Ponder Cloud, customerId is the
// real key and workspaceId becomes a string mirror of customers._id.
sessions: defineTable({
  ...,
  customerId: v.optional(v.id("customers")),
  ...,
})
  .index("by_customer_status", ["customerId", "status", "createdAt"]),

workers: defineTable({
  ...,
  customerId: v.optional(v.id("customers")),
  ...,
})
  .index("by_customer", ["customerId"]),
```

### 3.3 Query scoping

Every public Convex function on Ponder Cloud accepts a `clerkAuthToken` (Convex Auth wraps this) and resolves it to a `customerId` via `ctx.auth.getUserIdentity()` + a `customers` lookup. Then every read/write filters by that `customerId`. Self-host deployments skip the auth step (no `customers` table populated) and queries fall back to the legacy un-scoped path.

```ts
// helper used by every public function on Ponder Cloud
async function requireCustomer(ctx: QueryCtx | MutationCtx) {
  const identity = await ctx.auth.getUserIdentity();
  if (!identity) throw new Error("not authenticated");
  const orgId = identity.org_id; // Clerk org claim
  const customer = await ctx.db
    .query("customers")
    .withIndex("by_clerkOrg", q => q.eq("clerkOrgId", orgId))
    .first();
  if (!customer) throw new Error("customer not found");
  return customer;
}
```

### 3.4 Migration path

Self-host deployments and Ponder Cloud share the same schema. The difference is *which queries are deployed*. We split `convex/sessions.ts` → `convex/sessions.public.ts` (Cloud, requires auth) + `convex/sessions.ts` (self-host, no auth). Both are valid Convex functions; consumers import the one that matches their deployment.

For the SDK, `PonderClient` adds `apiKey` (Cloud only) — the API key is exchanged for a Clerk JWT at dispatch time:

```ts
new PonderClient({
  convexUrl: "https://cloud.ponder.dev",
  apiKey: "pk_live_…",      // optional; Cloud-only
});
```

When `apiKey` is set, the client routes to `*.public` mutations. When unset, it routes to the legacy un-scoped path (works against any self-host deployment).

## 4. Item 8: Hosted Ponder Cloud SaaS

The hosted offering at `cloud.ponder.dev`. This is mostly stitching: the runtime already exists, the workers already work, we add a dashboard + sign-up + provisioning glue.

### 4.1 Components

| Component | Stack | Purpose |
|---|---|---|
| **Convex deployment** | Single shared Convex instance | The "cloud" backend everyone shares. Multi-tenant via `customerId`. |
| **Dashboard** | Next.js + Clerk + tRPC (or just Convex queries from the React app) | Sign in, see sessions, watch live, manage workers, see usage |
| **Provisioner** | Node service on AWS Fargate | Turns "click button" into "container running with workspace creds". Wraps AWS Fargate API behind `IWorkerHost`. Modal stays the GPU-only backend for Holo3 inference (existing). |
| **Auth** | Clerk | Identity, orgs, JWTs |
| **Billing** | Stripe Billing + metered prices | Payment, invoices, plan changes (§5) |
| **Marketing site** | Same Next.js app | Pricing page, docs, sign-up CTA |

### 4.2 Sign-up flow

1. Customer clicks "Get started" on the marketing site
2. Clerk's `<SignUp/>` widget → Google/email
3. Clerk auto-creates an org for new users (single-user orgs by default)
4. On `org.created` webhook (Clerk → our API): create a `customers` row with `planTier: "free"`, `hostedQuota: 0`
5. Redirect to dashboard onboarding:
   - Step A: install Ponder.app (free tier — BYO only) **or** upgrade to enable hosted
   - Step B: copy `convexUrl` + click `ponder://configure?…` to wire local Mac
   - Step C: dispatch a starter task to verify

### 4.3 Worker provisioning ("Spin up hosted worker")

**Compute split:** AWS for general-purpose worker containers (the always-on Linux + Electron + Selkies stack), Modal for GPU-bound Holo3 inference. AWS is cheaper at idle and supports always-warm containers well; Modal is the right primitive for bursty $3.20/hr H100 calls. We use both at different layers, not one or the other.

A button in the dashboard. On click:

1. Frontend calls `POST /api/workers/spawn` with `{ customerId }`
2. Server checks `customer.hostedQuota` and counts existing hosted workers
3. If under quota: `ecs.runTask({ taskDefinition: "ponder-hosted:LATEST", launchType: "FARGATE", overrides: { env: { CONVEX_URL, WORKSPACE_ID, CLERK_JWT_FOR_BOOT } } })`
4. Container boots, registers as worker (we already wrote that code in item 7)
5. Dashboard polls until the worker shows up in Convex
6. Customer sees a "Hosted worker (US-East-1)" tile in their dashboard

The `IWorkerHost` interface keeps us swap-ready (Modal Container, Fly Machines) if AWS gets in the way later:

```ts
interface IWorkerHost {
  spawn(opts: { customerId: string; image: string; env: Record<string,string> }): Promise<{ machineId: string; region: string }>;
  destroy(machineId: string): Promise<void>;
  status(machineId: string): Promise<"booting"|"running"|"stopped"|"error">;
}
```

The default impl is `FargateHost`. `ModalHost` and `FlyMachineHost` slot in behind the same interface if we want to fail over.

### 4.4 Live session viewing

Each session row has `runtime: "hosted"` and a `claimedBy: workers._id`. The dashboard:

1. Subscribes to `sessions.steps` for the row (already works — `subscribe()` from PonderClient)
2. If `runtime === "hosted"` and `status === "running"`: render an `<iframe src="https://stream.ponder.dev/{sessionId}?jwt=..."/>` showing the Selkies stream from the claiming worker
3. If `runtime === "desktop"`: show the step transcript only (we don't have streaming for BYO Macs in v1.2; that's a v2 feature)

### 4.5 Plan tiers

Positioning: **PostHog / Firecrawl-style**. Generous free tier so devs can build something real before they ever hit a paywall, no credit card to start, self-serve upgrade, transparent overage pricing. The product is dev-tools-shaped — pricing follows the dev-tools playbook.

**Unit economics (our cost):**
- Per action: **~$0.003-0.008** (mostly Modal Holo3 inference + planner LLM)
  - Holo3 H100: ~3s @ $3.20/hr ≈ $0.0027
  - Planner LLM: ~$0.005 (Claude / H Company average) — drops to ~$0.0024 when batched across multiple actions, which most loops do
  - Convex + storage + egress: $0.0002
- Per hosted worker hour: **~$0.06** (Fargate 1vCPU/2GB ≈ $0.05 + Selkies egress at 720p ≈ $0.01)

| Tier | $/mo | Actions included | Overage | Hosted workers | Hosted hr | Retention | Support | CC required |
|---|---|---|---|---|---|---|---|---|
| **Free** | $0 | **1,000** | hard cap | 0 (BYO only) | — | 30 days | Community Discord | No |
| **Hobby** | $29 | 5,000 | $0.01/action | 1 | $0.15 | 30 days | Email, 48h | Yes |
| **Growth** | $199 | 50,000 | $0.006/action | 5 | $0.12 | 90 days | Email, next-business-day | Yes |
| **Scale** | $999 | 500,000 | $0.003/action | 20 | $0.10 | 365 days | Slack channel, 4h | Yes |
| **Enterprise** | custom (≥$2k) | custom | volume tiered | custom pool | volume tiered | configurable | SLA + dedicated CSM | Invoice |

**Free tier math.** 1,000 actions = ~33 real tasks (avg ~30 actions per task). Worst-case cost to us per free user who hits the cap: ~$8/mo. Realistically most free users use a fraction. Big enough to "build something" — the PostHog/Firecrawl bar — small enough that we're not eating $50/mo on every sign-up. This is the acquisition cost.

**Margin notes.**
- Hobby: $29 covers ~3,800 actions at our cost → 30% margin headroom on the included bucket; overage is 2-3× cost = healthy.
- Growth: $199 covers ~25,000 actions at our cost → margin tightens at scale on the bundled bucket but overage pays. Goal is to make this the median paying tier.
- Scale: bundled actions priced near cost; revenue from hosted-worker hours and feature gating (SAML, longer retention, priority support).
- Hosted worker hours are priced **3×** our infra cost across the board — covers networking, support overhead, and the fact that customers expect 99.9% uptime which costs more than just the Fargate bill.

**Free-tier gating** (i.e. what's not in Free):
- Hosted workers (paid tiers only)
- SAML SSO (Scale+ via Clerk's SAML add-on)
- Audit logs (Growth+)
- 24/7 support (Scale+)
- Custom retention (Enterprise)

What's **in** Free that PostHog/Firecrawl-grade dev-tools companies offer: all SDK features, all action verbs, dashboard, live session view (for BYO), step history, the CLI, public schema docs. No artificial feature crippling — the limit is volume, not capability.

**Annual discount.** 20% off when billed annually, same on Stripe Billing's subscription_data.billing_cycle_anchor. Standard Stripe webhook flow.

### 4.6 Sign-up funnel target

PostHog hit ~10k+ self-serve sign-ups/quarter once free tier was generous. Loose target for us at six months post-launch:
- 2,000 free sign-ups
- 5% paid conversion → 100 paying customers
- Of those: 80 Hobby ($29 = $2.3k MRR), 18 Growth ($199 = $3.5k MRR), 2 Scale ($999 = $2k MRR)
- ~$7.8k MRR at month 6

This isn't a financial plan, it's a sanity check that the tier math points somewhere positive. If we close even one Enterprise deal in months 4-6, it doubles MRR.

## 5. Stripe billing (out-of-scope #3 → in scope)

### 5.1 Schema

```ts
usageRollups: defineTable({
  customerId: v.id("customers"),
  periodStart: v.number(),                // unix ms, start of UTC day
  actionsCount: v.number(),
  hostedSeconds: v.number(),              // metered: hosted-worker uptime
  reportedToStripeAt: v.optional(v.number()),
}).index("by_customer_period", ["customerId", "periodStart"]),
```

A daily cron (`crons.ts`) reads `steps` for each customer for the prior UTC day, counts `kind: "action"` rows, and upserts a `usageRollups` row.

### 5.2 Reporting to Stripe

Use Stripe's [Billing Meter Events API](https://docs.stripe.com/billing/subscriptions/usage-based/recording-usage). One meter per metric:
- `actions_count`
- `hosted_seconds`

Daily cron iterates `usageRollups` rows where `reportedToStripeAt` is null, calls `stripe.billing.meterEvents.create(...)` for each, sets the timestamp. Idempotent on retry via `(customerId, periodStart, metric)` natural key.

### 5.3 Webhook handler

Inbound `/api/stripe/webhook` routes:
- `customer.subscription.created` → set `customers.planTier`, raise `hostedQuota`
- `customer.subscription.updated` → re-sync planTier + quota
- `customer.subscription.deleted` → drop to free tier; destroy hosted workers in excess of new quota
- `invoice.payment_failed` → flag the customer; dashboard shows banner; existing sessions complete but new dispatches reject

### 5.4 What we deliberately don't build

- **Self-serve plan downgrades that destroy hosted workers** — manual support ticket for v1.2; trivial to script later
- **Promo codes / coupons** — Stripe handles in checkout; we don't model in our DB
- **Annual billing toggle** — Stripe handles; not in our schema

## 6. Modal Holo3 robustness (out-of-scope items #4, #6, #7 — partial)

Three Modal items were on the out-of-scope list. We tackle two of three:

### 6.1 GPU scheduling beyond Modal's built-in (#4) — **defer**

Modal's container concurrency knobs (`@app.cls(concurrency_limit=N)`) cover the 90% case. Custom GPU scheduling means writing our own queue + autoscaler, which is multi-week work and irrelevant until we have >100 concurrent hosted workers. Punt to v2.

### 6.2 Per-workspace Modal secret rotation (#6) — **ship**

Today, `MODAL_TOKEN_ID` and `MODAL_TOKEN_SECRET` are baked at deploy time. For Ponder Cloud, every customer's Modal calls go through *our* Modal account using *our* tokens — fine, but it means we need rotation hygiene:

- Add `cron: rotateModalTokens()` running monthly
- Reads expected token from a managed secret store (we'll use Doppler — already supports rotation hooks)
- New token deployed; `convex deploy` replays all functions with new env
- Old token revoked 24h later via Modal API

This is operations, not feature work. ~2 days.

### 6.3 Multi-region Modal deployments (#7) — **defer**

**Region for v1.2: US-East only.** Everything (AWS Fargate, Modal, Convex, Selkies) lives in `us-east-1` / equivalent. EU customers tolerate ~80-120ms RTT for now; the agent loop is not latency-sensitive at human-perception scale. Adding EU (Frankfurt) is the first multi-region project once US-East revenue justifies the doubled infra cost — probably v1.4 or when we have 3+ EU customers.

Modal multi-region session-affinity is the harder part and stays deferred to v2.

## 7. Sub-second cancellation (out-of-scope #5) — **partial**

Today, cancellation works via polling: when `sessions.status === "cancelled"`, the agent loop notices at the next `await checkCancelled()` boundary, which can be up to a few seconds depending on what verb is mid-flight (LLM call, screenshot, etc.). Sub-second propagation requires push (server-sent events or a Convex subscription) inside the agent loop.

**v1.2 ship:** add a Convex subscription inside the agent loop that toggles a local AbortController on `status: "cancelled"`. Verbs check `signal.aborted` between primitives. This gets us ~50-200ms cancel typical, not sub-second guaranteed but much better than now. Real sub-second SLA is a v2 problem.

~3 days.

---

## 8. Replay viewer (out-of-scope #2) — **defer**

The dashboard's session detail page shows the step transcript. That's "replay" enough for v1.2: scroll through steps, click thumbnails to enlarge screenshots. A "scrub through frames" video-player UX is what was originally on the wishlist; that's a v2 polish.

Save effort for v2 by storing screenshot order + timestamps cleanly now (already do — `steps.createdAt`).

---

## 9. Plugin system (out-of-scope #8) — **deferred to v2**

A plugin system means committing to a public extension contract: registering verbs, calling into the planner, sandbox semantics, versioning, security. That's an architectural commitment, not a feature. We don't ship public extension points until we've used Ponder ourselves enough to know what verbs *should* exist.

Internal "verb extensions" (like the file-system verb the Modal people asked about) are easy: just add them to the agent's verb registry. No formal plugin system needed yet.

## 11. Decisions locked in

These were open questions at draft time; all seven are now answered. Recording them here so the implementation never has to re-litigate.

| # | Question | Decision | Rationale |
|---|---|---|---|
| 1 | Auth | **Clerk** | Saves ~2w of org modeling; SAML add-on covers future enterprise; JWT shape is portable if we ever migrate to Convex Auth |
| 2 | Hosted-worker host | **AWS Fargate** (general compute) + **Modal** (Holo3 GPU) | AWS is the right primitive for always-warm Linux containers; Modal stays where it already shines |
| 3 | Display streaming | **Selkies WebRTC** + coturn TURN fallback | Sub-100ms RTT matters because customers are watching the agent click; x264 software encoding sidesteps Fargate's no-GPU constraint |
| 4 | Billing model | **Stripe metered** on actions + hosted-hours | Aligns price with cost; customers pay for what they use; overage smooths the tier-jump cliffs |
| 5 | Region | **US-East-1 only** at launch | EU added in v1.4 / when 3+ EU customers ask |
| 6 | Pricing | **Free $0 / Hobby $29 / Growth $199 / Scale $999 / Enterprise custom** | See §4.5 for full breakdown including overage rates and hosted-worker hourly pricing |
| 7 | Free tier size | **1,000 actions/mo** | PostHog/Firecrawl-level — devs can build something real before hitting the wall. ~$8/mo max cost per free user; sustainable acquisition expense |

The single most important call here is **#7**: a stingy free tier kills the dev-led-growth motion that PostHog and Firecrawl proved out. 1,000 actions is the floor; we move it up if real usage data says we should.

## 12. What ships at the end

A user can:
- Sign up at cloud.ponder.dev with Google or email
- Install Ponder.app on their Mac and click a `ponder://` link to wire it
- (Or, if on Starter+) click "Spin up hosted worker" in the dashboard and watch a Linux Ponder spin up in ~30s
- Dispatch tasks via `new PonderClient({ apiKey: "pk_live_…" })`
- See live session steps + (for hosted) a live screen stream
- Get billed monthly via Stripe
- Cancel a session and see it stop within ~200ms

Everything else from the original lists either ships in v1.3+ or stays out of scope permanently.
